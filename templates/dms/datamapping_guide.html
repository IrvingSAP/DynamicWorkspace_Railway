Cómo funciona Data Mapping

FilePipe / Data Mapping convierte archivos de un formato a otro sin escribir código: defines el archivo de origen, el de destino, el mapeo campo a campo y las reglas; luego subes un archivo, ejecutas la transformación y descargas el resultado. Cada transformación vive en un proyecto reutilizable de tu compañía.

Esta página describe el flujo completo. En cada pantalla del producto también hay ayuda contextual (botón ? / Ayuda) con el detalle de esa vista.

Ruta rápida

  1. Crea un proyecto en Data Mapping (visibilidad y miembros).
  2. Completa Origen → Destino → Mapeo → Reglas y publica una versión.
  3. Sube el archivo, ejecuta la transformación y revisa el historial.

Qué problema resuelve

Para qué sirve: unificar conversiones recurrentes de archivos (TXT posicional, CSV, Excel y similares) en proyectos configurables, en lugar de scripts sueltos o trabajo manual en hojas de cálculo.

Motor en resumen

  1. Leer y parsear el archivo de entrada según el perfil de origen.
  2. Aplicar mapeo de campos y reglas de transformación.
  3. Serializar la salida según el perfil de destino.

Puntos relevantes

  • Todo ocurre dentro de tu compañía; solo ves proyectos donde eres miembro o los públicos de la organización.
  • La definición se versiona: trabajas en un borrador y luego publicas una versión activa para ejecutar.
  • Origen y destino usan asistentes de 6 pasos cada uno.

Fase A — Acceso al proyecto

PA

Para qué sirve: crear el contenedor del proceso, decidir quién lo ve y quién puede editarlo o ejecutarlo.

Pasos

  1. Crear proyecto — nombre, slug y descripción desde «Nuevo» en el listado.
  2. Visibilidad — privado (solo miembros) o público de compañía (visible en la organización).
  3. Miembros — asignar roles PA / ED / CO / GE desde el hub del proyecto.

Puntos relevantes

  • El creador suele quedar como PA.
  • Sin membresía adecuada no podrás editar origen/destino ni publicar.
  • Desde el hub ves el stepper de fases A → B → C.

B

Fase B — Definición técnica

PA ED

Para qué sirve: describir cómo se lee el origen, cómo se escribe el destino, cómo se conectan los campos y qué transformaciones se aplican. El orden recomendado es el del hub.

Orden en el hub

  1. Origen — tipo de archivo, encoding, layout, campos (6 pasos) y publicar perfil.
  2. Destino — mismo esquema de asistente para la salida (puedes cargar campos desde origen).
  3. Mapeo — enlazar campos origen → destino; los obligatorios del destino deben quedar cubiertos.
  4. Reglas — transformaciones declarativas sobre el pipeline (mayúsculas, recortes, valores fijos, etc.).
  5. Publicar — congela el borrador en una versión activa usable en ejecución.

Puntos relevantes

  • Hasta publicar, trabajas sobre un borrador (p. ej. «borrador vN»).
  • No podrás publicar si faltan origen, destino, mapeo u reglas completos.
  • Usa la ayuda «?» de cada paso del asistente para el detalle de campos y opciones.

C

Fase C — Ejecución

PA ED GE

Para qué sirve: aplicar la versión publicada a un archivo real, obtener la salida y auditar lo ocurrido.

Pasos

  1. Archivo — subir o seleccionar el fichero de entrada (intake).
  2. Ejecutar — preview opcional y transformación completa.
  3. Descarga — archivo de salida (y reporte si aplica).
  4. Historial — trabajos previos del proyecto para reconsulta.

Puntos relevantes

  • La ejecución usa la versión activa publicada, no el borrador en edición.
  • Si cambias origen/destino/mapeo/reglas, vuelve a publicar antes de ejecutar en producción.
  • El hub marca «Archivo + transformar» cuando el intake está listo.
